# 审计总览

## 结论

当前项目的主要问题不在某一行业务代码，而在三个基础面：

1. 共享层是空壳，目录结构和类型边界不一致。
2. 前端模板代码已经把 i18n、组件复用和状态边界做窄了。
3. 构建与校验链不可信，CI 很难成为真正的交付防线。

## 当前优先级

### P1. 先修基础设施，再谈业务扩展

- `packages/api`、`packages/core`、`packages/types` 目前只有 `package.json`
- `tsconfig.base.json` 已经声明路径映射到 `packages/*/src`
- 两个 app 都已依赖这些 workspace 包

这会制造一个危险假象：看起来已经有“共享层”，实际上共享层并不存在。

### P1. 先做 i18n 边界，再做页面扩展

目前 `apps/mobile` 和 `apps/miniapp` 的文案直接写死在组件中。继续这样推进，后续任何一个需求都会被放大：

- 多语言
- 品牌换肤
- 埋点文案统一
- A/B 实验
- 跨端复用

### P1. 先让检查链可信

根脚本声明了 workspace build，但子包没有形成一致的 `build` / `typecheck` / `lint` 约定。再叠加这次实际遇到的 `pnpm` 签名校验失败，意味着当前仓库缺少一个稳定的交付闸门。

## 先落地的三步

### 第一步：做实共享层

- 为 `packages/core`、`packages/types`、`packages/api` 创建真实 `src/`
- 明确导出边界与 `exports`
- 把多端共用的文案、类型、常量、请求契约收进共享层

### 第二步：收紧前端边界

- 引入统一的 locale/messages 入口
- 修正虚假的主题覆盖 API
- 把页面级静态文案移出 JSX

### 第三步：补齐交付门禁

- 根目录统一 `lint` / `typecheck` / `build`
- 子应用统一声明对应脚本
- 后续再引入最小 smoke test

## 可取之处

当前仓库并非一无是处。`scripts/start.mjs` 把多应用启动入口统一了，交互脚本也比较干净，说明工程入口有一定整理意识。但它只解决“怎么启动”，没有解决“怎么可靠交付”。
